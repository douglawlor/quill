# Skill Quill Pack (.sqp) — Design Proposal

Phase 3 (Prompt Library, Ask AI, .pqp) is shipped. This document covers the next
frontier: AI skills written in plain text that QUILL interprets as multi-step
workflows.

---

## The idea in one sentence

A `.sqp` file is a markdown document where the headings are steps, the body is
natural-language instruction, and special fenced blocks declare parameters,
conditions, and outputs — so an AI skill reads like a recipe and runs like a
program.

---

## Why plain text?

Prompt packs (`.pqp`) are flat: a name, a category, and one instruction. That
covers most everyday tasks. But some AI work is inherently multi-step:

- Research a topic, then draft a paragraph using what was found.
- Read the document title, then suggest three alternative titles.
- Check whether the selection is a question or a statement, then take a
  different branch based on the answer.
- Summarise a long passage in chunks, then combine the summaries.

Encoding this in JSON means writing a DSL nobody can read or author without
tooling. Writing it in Markdown means any screen-reader user, any writer, any
developer, can open the file in QUILL itself and understand — or edit — every
step immediately.

The hypothesis: if the file format is the same tool the user already lives in,
the barrier to authoring disappears.

---

## File anatomy

A `.sqp` file is a Markdown document with a YAML front matter block.

```markdown
---
schema: quill.skill/1
name: Research and Draft
description: Researches a topic from the selection and drafts a paragraph.
author: QUILL Project
version: 1.0.0
parameters:
  - name: tone
    label: Tone
    type: choice
    choices: [formal, conversational, neutral]
    default: neutral
---

# Step 1: Extract topic

Identify the main topic from the following text. Return only the topic as a
single noun phrase — no sentence, no punctuation.

```input
{selection}
```

# Step 2: Research

List five factual claims about "{step1.output}" that would support a short
explanatory paragraph. Number each claim. Keep each claim to one sentence.

# Step 3: Draft

Write a single cohesive paragraph about "{step1.output}" that weaves in the
facts from Step 2. Tone: {parameters.tone}. Length: 80–120 words.

```output
format: text
label: Drafted paragraph
```
```

That is the complete skill. QUILL reads it, identifies three steps, runs them
in sequence, passes outputs forward as variables, and shows the final result in
the standard AI Response dialog.

---

## Anatomy of a step

Each `# Heading` at depth 1 is a step. The body of the step is the prompt sent
to the model. Step prompts are plain prose with variable interpolations.

**Available variables in step prompts.**

| Variable | Value |
| --- | --- |
| `{selection}` | Current editor selection (or full document if nothing selected) |
| `{document}` | Full document text |
| `{title}` | Document title |
| `{stepN.output}` | Output text produced by Step N |
| `{parameters.name}` | Value of a declared parameter |
| `{clipboard}` | System clipboard text |

**Special fenced blocks inside a step.**

````markdown
```input
{selection}
```
````

An `input` block appends literal text to the prompt. Useful when you want to
keep the step prose clean and the data clearly separated.

````markdown
```condition
if: "{step1.output}" contains "question"
then: step3
else: step4
```
````

A `condition` block is evaluated before the step's prompt is sent. `then` and
`else` name the next step to jump to. If omitted, execution falls through to
the next heading in document order.

````markdown
```output
format: text | list | json
label: Human-readable label shown in the result dialog
accept_into: selection | clipboard | none
```
````

An `output` block on the *last* step controls how QUILL presents the result.
`accept_into: selection` replaces the editor selection when the user clicks
Accept. `accept_into: clipboard` copies to clipboard. `none` (default) shows
the result read-only.

---

## Parameters

Declared in front matter. QUILL renders a small parameter dialog before running
the skill if any parameters are declared. All parameter types are accessible:

| Type | UI rendered |
| --- | --- |
| `text` | Single-line TextCtrl |
| `multiline` | Multi-line TextCtrl |
| `choice` | wx.Choice populated from `choices` list |
| `bool` | wx.CheckBox |
| `number` | TextCtrl with numeric validation |

Parameters default to their `default` value; the parameter dialog may be
skipped if all parameters have defaults and the skill sets `skip_params: true`
in front matter.

---

## Execution model

QUILL's `.sqp` runner:

1. Parses the front matter (YAML) and validates against the skill schema.
2. Collects parameters (shows dialog if needed).
3. Resolves the step order from heading sequence, modified by any `condition`
   blocks encountered at runtime.
4. Executes each step in turn:
   a. Interpolates variables into the prompt text.
   b. Sends to the configured model (uses `ai_prompt_default_model`, falls back
      to `ai_chat_default_model`).
   c. Stores the response as `stepN.output`.
   d. Evaluates any `condition` block to determine the next step.
5. After the final step, presents output via the AI Response dialog (or applies
   it per the `output` block declaration).
6. Every step send is announced ("Running step 2 of 3...") and each step result
   is available for review in a step-by-step transcript accessible from the
   result dialog.

Each step is a discrete undo point. If the user closes mid-skill, nothing has
changed in the document.

## Composability

A step can invoke a named prompt from the Prompt Library:

````markdown
# Step 2: Clean up

```use-prompt
name: Improve Clarity
input: {step1.output}
```
````

This delegates to the "Improve Clarity" built-in prompt (or the user's override
of it) rather than duplicating the instruction. Skills stay thin by reusing the
prompt library.

A skill can also call another skill:

````markdown
# Step 3: Deep dive

```use-skill
name: Research and Draft
parameters:
  tone: formal
input: {step2.output}
```
````

Nested skills are bounded to two levels deep to keep execution predictable.

---

## The `.sqp` file as a QUILL document

Because `.sqp` is Markdown, a user can open it in QUILL's editor, read every
step with their screen reader, and edit any instruction directly. There is no
GUI skill builder, no node graph, no DSL reference card to memorise. The skill
is the document.

This also means all existing QUILL editing features work: search, bookmarks,
word count, abbreviation expansion, copy tray. A power user can maintain a
collection of `.sqp` files in a Watch Folder and have QUILL keep them indexed.

---

## Sharing skills

Skills are shared as `.sqp` files — a plain Markdown file with a `.sqp`
extension. Import via the Prompt Library dialog (extend Import to accept both
`.pqp` and `.sqp`). When a `.sqp` is imported it appears in a "Skills" section
of the library separate from flat prompts.

A Quillin may also ship `.sqp` files in its package. QUILL discovers them
alongside `prompts.json` at library load time.

## Example: "Accessible Rewrite"

```markdown
---
schema: quill.skill/1
name: Accessible Rewrite
description: Rewrites selected text for plain-language accessibility.
version: 1.0.0
parameters:
  - name: reading_level
    label: Target reading level
    type: choice
    choices: ["Grade 6", "Grade 8", "Grade 10", "No target"]
    default: "Grade 8"
---

# Step 1: Analyse

Review the following text for accessibility issues. List the issues as a
numbered list. Focus on: sentence length, passive voice, jargon, abstract
nouns, and complex sentence structure. Be concise — one issue per line.

```input
{selection}
```

# Step 2: Rewrite

Rewrite the text below to fix the issues identified in Step 1. Target reading
level: {parameters.reading_level}. Preserve all factual content. Do not add
new information.

Original text:
{selection}

Issues to fix:
{step1.output}

```output
format: text
label: Rewritten text
accept_into: selection
```
```

The user selects a paragraph, runs "Accessible Rewrite", chooses a reading
level, reads the two-step analysis and rewrite, and presses Accept to replace
the selection. The whole interaction is keyboard-only, screen-reader
compatible, and fully undoable.

## Open questions (to resolve before implementation)

1. **Model per step.** Should individual steps be able to specify a model
   override (e.g. a cheaper model for analysis steps, a stronger one for
   creative output)? Adds power but complicates the parameter dialog.

2. **Streaming.** Should each step stream its response progressively into an
   intermediate read-only buffer, or wait for the full response before
   proceeding? Streaming is more responsive but harder to pass as a variable.

3. **Step transcript access.** Should the result dialog show a collapsible
   "Step transcript" listing each step's prompt and response? Useful for
   debugging skills; potentially verbose for users who just want the result.

4. **Skill categories.** Should skills have categories (like prompts) or a
   separate tagging system?

5. **Parameter pre-fill from context.** Should QUILL try to infer parameter
   values from the document (e.g. detect document language and pre-select it as
   a parameter)? Opt-in per parameter via `infer: true`?

6. **Condition syntax.** The `condition` block above uses a simple
   contains/equals check. Is that enough, or do skills need a richer expression
   language? A safe subset: `contains`, `starts_with`, `ends_with`, `equals`,
   `length_gt`, `length_lt`, `is_empty`.
